# DD-045: 家族の最終形・monorepo 段階戦略・マニフェスト実装ルール

**Date:** 2026-04-24
**Sessions:**
- [Family Ideal & Monorepo](../sessions/2026-04-24-tramli-family-ideal-and-monorepo.md)
- [Coverage & Migration](../sessions/2026-04-24-tramli-family-coverage-and-migration.md)

**Gap:** G10, G11, G12

## Decision

tramli ファミリーの最終形、monorepo 運用戦略、マニフェストの実装詳細を一括で凍結する。

### Part 1: 家族の最終形の再定義

- **「あらゆる部分をカバー」は「盲点ゼロ」に再解釈**する
  - ❌ 家族が全能ツールとして全部を書く
  - ✅ 家族が主要領域をカバー、未カバー領域は外部委譲として明示
- **3 軸（検証可能・再現性・宣言的）は独立次元**。同じレイヤーで全部を追わない
  - 検証可能 + 宣言的 = 家族が担う
  - 再現性 = repo 次元（monorepo / Nix / Bazel 等）に委譲
- MDA/UML の挫折（3 軸同時追求）を教訓とし、家族は意識的に回避する

### Part 2: monorepo 段階戦略

umbrella repo `tramli-family` を新設するが、段階的に移行する。

| Step | 内容 | 判断 |
|------|------|------|
| **1** | umbrella repo 新設、マニフェスト / DDs / sessions / CI / 依存図のみ配置 | **本 DD で採用** |
| **2** | tramli-reactive を umbrella 内で新規作成 | 設計時期が来たら着手 |
| **3** | 既存 repo (tramli, tramli-appspec) を umbrella に統合 | **強制しない**。必要性が発生してから判断 |

### Part 3: 既存 DGE 文書の移行（時系列分割）

- **2026-04-24 以前**: tramli repo の `dge/` にそのまま残す（移行コストゼロ、リンク無傷）
- **2026-04-24 以降**: umbrella repo `tramli-family` へ配置
- 2026-04-24 当日の 6 session + DD-042/043/044/045 は **umbrella repo の創設文書**として扱う
- tramli repo の `dge/` に「家族全体の議論は umbrella repo へ」という案内を残す

### Part 4: マニフェスト実装ルール（DD-044 の補強）

- **カバレッジ表粒度**: 本体は粒度 B-2（5 行：ドメイン / 通信 / 永続化 / UI / 運用）
- **階層化**: L1 マニフェスト（5 行）/ L2 付録（7 行）/ L3 個別 README（詳細）
- **3 軸独立性の記述**: 抽象論ではなく具体事実（「2 軸担当、1 軸委譲」）。非ゴール章 5.1 冒頭 1 行で MDA/UML 挫折に触れる
- **委譲先ツールには日付を付記**（例: `React（代表例、2026-04-24 時点）`）— 古くなっても壊れない文書設計

### Part 5: カバレッジ表更新メカニズム

| トリガー | 内容 | 承認 |
|---------|------|------|
| T1 | 新メンバーが追加された | DD 必須 |
| T2 | 既存メンバーの担当範囲が変わった | DD 必須 |
| T3 | 委譲先ツールが変わった | 軽量更新 + 日付付記（DD 不要） |

### Part 6: 新メンバー追加基準（マニフェスト付録に明記）

1. 独立パッケージとして build / test できる
2. 依存ツリーがルート（tramli）を向いている（DAG）
3. 既存メンバーの責務と重複しない（カバレッジ表マトリクスで重複セルが出ない）
4. 宣言的 DSL または型付き API を提供する（家族の美学）
5. 提案 DGE session で承認される

## Rationale

- 「あらゆる部分」を全能ツールとして追う試みは歴史的に失敗する（MDA/UML）。
  盲点ゼロ + 外部委譲明示で実質的に同じ価値を提供できる（深澤視点）
- 3 軸を同じレイヤーで追わない = 関心事の分離。再現性を repo 次元に委譲することで、
  家族は検証可能+宣言的に集中できる
- monorepo の段階的採用は、既存 repo の独立性を壊さずに
  家族の集合としての見え方を獲得する中間解
- 時系列分割は既存リンクを無傷で保ちつつ、意味論的に正しい新居を用意する
  （ヤン視点での最小コスト解）
- 委譲先ツールの日付付記は「古くなっても壊れない」文書設計（深澤視点）
- 新メンバー追加基準の 5 項目目「DGE 承認」により、家族拡張プロセスが自己完結する

## Alternatives considered

- 完全 monorepo（全 repo 即時統合）→ 却下（3 言語 + 独立リリース + 移行コスト）
- 現状維持（分散 repo）→ 却下（家族という集合が見えない、CI 統合不可）
- 全 DGE 文書を umbrella に移す（M2）→ 却下（既存リンクが壊れるコスト）
- DGE 文書を tramli repo に残す（M1）→ 却下（家族文書が片メンバー repo にある意味論的違和感）
- カバレッジ表を粒度 B（7 行）で書く → 却下（マニフェスト 2 ページに収まらない）
