# DD-044: ファミリーマニフェストの 3 精度ルール（依存ツリー / DD 同期 / 期待値調整）

**Date:** 2026-04-24
**Session:** [Manifest Integrity](../sessions/2026-04-24-tramli-manifest-integrity.md)
**Gap:** G7, G8, G9

## Decision

DD-043 で採用した tramli ファミリーマニフェストを書くときの 3 精度ルールを定める。

### Rule 1: 依存ツリー制約の精密化（G9 解消）

- マニフェストで「ツリー制約」と書かず、**「依存ツリー制約」** と書く
- 対象は **パッケージ import 依存グラフ**（Maven / npm / Cargo 等）のみ
- データフロー / event / callback / メッセージ送受信は対象外
- マニフェスト内では **2 枚の Mermaid 図** を使い分ける:
  - 図 A: 依存ツリー（DAG、import 依存）
  - 図 B: データフロー（双方向可、実行時の情報流）

### Rule 2: マニフェスト ⇄ DD 同期メカニズム（G8 解消）

3 段構え:

1. **S1: 本文埋め込み** — マニフェスト本文中の主張に該当 DD 番号を付す
   （例: `...完結させる [DD-042]`）
2. **S2: 末尾根拠一覧** — マニフェスト末尾に「本マニフェストの根拠: DD-042, DD-043, DD-044」を置く
3. **CI 軽量チェック** — 簡易スクリプトでマニフェスト内の DD 番号と DD 側の逆リンク整合を検証

自動生成（S3）は採用しない（人間が書くニュアンスを殺すため）。

### Rule 3: 期待値調整の二重強調（G7 解消）

- **冒頭 3 行** で negative definition を使う:
  > tramli ファミリーは、アプリケーションの**一部分を構造的に検証できる**
  > 独立した宣言系の集合です。UI レンダリング、通信プロトコル、
  > インフラ配置などは別ツールに委譲します。
- **非ゴール章（5 章）** で詳細を 3 階層（ファミリー全体 / 各メンバー / 検討して却下）で列挙
- 委譲先ツールは **代表例 1 つのみ** 挙げる（技術選定をユーザーに委ねる）

## Rationale

- G9 は設計矛盾ではなく **用語の曖昧さ** が原因だった。「ツリー」が依存 / データ流 / callback の区別なく使われていた
- G8 は軽量 CI チェックで人間ミスを自動検出しつつ、文書の表現力は人間に残すのが最適
- G7 は冒頭と非ゴール章の二重強調により、読者が 2 度「これではない」と聞くことで誤読を防ぐ
- 3 Gap いずれも **骨子の設計変更を要さない**。書き方の精度の問題のみ（Harel 視点）
- これらは設計フェーズの **最後の DD**。次は本文執筆フェーズ

## Alternatives considered

- S3（マニフェスト自動生成）→ 却下（人間が書くニュアンスが失われる）
- 「ツリー制約」のまま曖昧に残す → 却下（読者が誤読し、ファミリー設計への不信を招く）
- 期待値調整を非ゴール章のみに集約 → 却下（冒頭で期待値が定まらないと誤読が始まる）
