## Scene 1: 「理想形」の定義 — 3 語の意味を詰める

**先輩（ナレーション）**: 「検証可能で再現性のある宣言的」— 耳障りが良いが、中身を詰めないと空論になる。ハレル教授とミルナー教授から定義を始める。

---

**🔬 ハレル教授**: 3 語を別々に定義します。

```
検証可能 (Verifiable):
  宣言した仕様に対する正しさを、機械的に確認できる。
  形式手法: type checking, model checking, property testing, proof.

再現性 (Reproducible):
  同じ入力から同じ出力が必ず得られる。
  build, execution, verification の各フェーズで成立すべき。

宣言的 (Declarative):
  「何を」書くだけで、「どのように」は記述しない。
  制御フロー (how) ではなく目的 (what) を書く。
```

tramli は **宣言的 + 検証可能（ビルド時）** を小さな領域で満たしています。再現性は偶然的に成立（純粋関数 + FlowStore 抽象）。

**🧮 ミルナー**: 理論的に言えば、3 語は **独立な次元** です。

```
         宣言的
          |
検証可能 ─┼─ 再現性
          |
       (実装技術)
```

3 軸それぞれに異なる先行研究があります:

```
検証可能:   Harel Statecharts, Type systems (Strom 1986), Model checking (SPIN, TLA+), Proof (Coq, Agda, Lean)
再現性:     Nix/NixOS, Bazel hermetic builds, content-addressed storage (Git, IPFS)
宣言的:     Prolog, SQL, Datalog, HTML, CSS, YAML, Dhall
```

**☕ ヤン**: *紅茶を置く。* 3 軸全部を全領域で満たすのは **不可能** です。先行研究を見ても、どれも一部しか成功していない。**どこまで諦めるかの判断**が先にあるべきです。

**👤 今泉**: えっと、tramli が「一部で成功している」って、具体的にはどこですか？ どの3軸にどの程度？

**🎨 深澤**: *静かに考える。* 僕の目には、tramli は **「小さな領域で 3 軸を全部揃えた exemplar」** に見えます。workflow + data-flow という限定的な領域で、宣言的に書ける、ビルド時に検証される、純粋関数で再現する。**この 3 軸の整合を小さく実現した例**として成立している。問題は **規模が小さい** ことです。

→ Gap 発見: **「理想形」の 3 軸は独立次元。全部を同じ強度で満たすのは不可能** — 先行研究の全ツール（Coq, Nix, Dhall 等）も何か 1-2 軸を妥協している。tramli 家族も妥協点を明示する必要がある。

→ アイデア: **3 軸の妥協マップを作る** — 各家族メンバーが 3 軸のどこに重みを置くかを明示する。例えば tramli-core は検証可能を優先、tramli-reactive は宣言的を優先、など。

## Scene 2: 同系譜の先行例 — 「理想を追った者たち」

**先輩（ナレーション）**: 同じビジョンを持った先行プロジェクトが何を達成し、何に挫折したかを見る。

---

**🧮 ミルナー**: 「検証可能・再現性・宣言的」を志向した歴史的試みを列挙します。

```
1. UML / MDA (Model-Driven Architecture, 2001)
   目標: 図から全コード生成、プラットフォーム独立
   結果: コード生成は部分的に成功、検証は不十分で衰退

2. Coq / Agda (1984-)
   目標: 定理証明による完全検証
   結果: 理論的には完全、実用コストが高すぎて一般化せず

3. NixOS (2003-)
   目標: システム全体の再現性
   結果: 再現性は成功、学習曲線と表現力の限界が課題

4. Dhall (2017-)
   目標: 検証可能な設定 DSL
   結果: 小規模成功、主流化には至らず

5. TLA+ (1999-)
   目標: 宣言的仕様の検証
   結果: Amazon 等で採用、コードとの統合は別問題

6. Bazel (2015-)
   目標: 再現性のあるビルド
   結果: Google 内部成功、外部採用は段階的
```

**パターン**: **1 軸または 2 軸に集中したものが成功**。3 軸全部を狙った UML/MDA は挫折しました。

**🔬 ハレル教授**: Statechart 自身がその一例です。私は検証可能 + 宣言的を狙いましたが、再現性（実行環境依存しない実行）は諦めました。完璧な Statechart 実行系は未だ存在しません。

**☕ ヤン**: *紅茶を啜る。* つまり歴史的教訓は **「3 軸全部を追うと失敗する」** ですね。tramli 家族も **1-2 軸に集中** すべき。では何を選ぶか。

**🎨 深澤**: *そう思う。* tramli の今の強みは **「検証可能 + 宣言的」** です。再現性は副作用として得ている。ここを守って拡張するのが自然。再現性は Nix や Bazel に任せて、家族は 2 軸に集中する方が美しい。

**👤 今泉**: えっと、**「再現性を捨てる」と言うより「再現性は別レイヤーに任せる」** ってことですか？ それなら monorepo の話と繋がりますね。Bazel を使えば再現性は repo 構造として得られる。

**🧮 ミルナー**: *頷く。* 今泉君の整理は正確です。**3 軸を同じレイヤーで全部やるのではなく、各軸を得意なレイヤーに割り当てる**。これは **関心事の分離** の応用です。

→ アイデア: **tramli 家族は「検証可能 + 宣言的」の 2 軸に集中** — 再現性は Nix / Bazel 等の外部レイヤーに委譲
→ アイデア: **3 軸の担当レイヤーを明示する**:
```
  検証可能: tramli 家族（Statechart + data-flow + session type）
  宣言的:   tramli 家族（DSL, ApplicationSpec）
  再現性:   Nix / Bazel / hermetic CI（repo 構造レベル）
```

→ Gap 発見: **MDA/UML 挫折の教訓が tramli 家族にどう活きるか未定義** — 3 軸全部を追ったから失敗した、という仮説の確認が必要。家族マニフェスト 5 章（非ゴール）に「3 軸を同じレイヤーでやらない」を入れるべき。

## Scene 3: tramli の成功要因は拡張可能か — 「あらゆる部分」は現実的か

**先輩（ナレーション）**: tramli が小さく成功した要因を分析し、それが「あらゆる部分」に拡張できるかを問う。

---

**🔬 ハレル教授**: tramli が検証可能 + 宣言的を両立できた理由を分析します。

```
成功要因:
  1. 領域が狭い (workflow + data-flow only)
  2. 型が軽量 (branded string for FlowKey)
  3. 副作用が分離 (processor は純粋であるべき契約)
  4. Build-time validation が決定的 (動的チェック不要)
  5. Flat enum (DD-021) により検証のエッジケースを最小化
```

これらのうち **1 (領域狭い) は拡張の敵** です。領域を広げるほど、2-5 の条件が崩れやすくなります。

**🧮 ミルナー**: 正確です。π-calculus も同じ問題に直面しました。小さな拡張でも表現力と検証可能性のトレードオフが発生します。**「あらゆる部分をカバー」を文字通り追うと tramli は tramli でなくなります**。

**☕ ヤン**: *紅茶を置く。* じゃあ **「あらゆる部分」は夢** として諦めましょうか？

**🎨 深澤**: *しばらく黙る。* …僕はユーザーの言葉を別の意味に読みました。**「あらゆる部分」は全能を意味しない**のではないか。各部分に **「家族の中の誰かが担当する」** 構造があれば、ユーザーから見て「全部カバーされている」と感じられる。深澤的に言えば **「盲点がない状態」** です。

```
ユーザーが知覚する「カバー」:
  × 1 ツールが全てを書ける (不可能)
  ○ 家族の誰かが各領域を担当し、境界が明示され、盲点がない
```

**👤 今泉**: えっと、**つまりユーザーが「あれはどうすればいい？」と聞いたとき、必ず「家族の誰々が担当」と答えられる状態** ってことですか？

**🎨 深澤**: *そう。* それが理想形。盲点ゼロ。

**🔬 ハレル教授**: *頷く。* それは実現可能な理想です。**「家族のカバレッジ表」** を明示すればいい。各領域（UI / 通信 / 永続化 / 認可 / 配置 / 監視 / 配線 / ...）について、家族のどのメンバーが担当し、どの程度の検証強度を提供するかを表にする。

**🧮 ミルナー**: 圏論的には **「家族 = 圏、領域 = 対象、メンバー = 対象の供給源」** という構造です。圏のカバレッジが全ての対象を覆うなら、ユーザーから見て「全部カバーされている」と言える。

**☕ ヤン**: *微笑む。* 結局 **マニフェストの 3 章（責務境界）マトリクス** の拡張版ですね。あれに「未カバー領域」列を足せば、盲点を可視化できる。

→ アイデア: **「家族のカバレッジ表」をマニフェスト 3 章に追加** — 主要領域（UI / 通信 / 永続化 / 認可 / 配置 / 監視）× 家族メンバー、セルは担当有無と検証強度
→ アイデア: **「未カバー領域」を明示する列を用意** — 未カバーなら「外部ツールに委譲: 代表例 X」と書く。盲点をゼロにする = 未カバー領域も明示することで盲点を潰す
→ Gap 発見: **「あらゆる部分」= 家族が全部書く、ではなく「盲点がゼロ」（外部委譲含む）** という再定義が必要。マニフェスト冒頭の期待値調整にこれを追加すべき

## Scene 4: monorepo の意味 — 「再現性」と「検証可能性」の repo 次元

**先輩（ナレーション）**: 技術的問いに戻る。家族を monorepo にまとめるべきか。3 軸の議論を踏まえて判断する。

---

**🎨 深澤**: repo 構造もデザインです。**「手に取れる形」** という観点から見ます。現在:

```
現状 (分散):
  github.com/opaopa6969/tramli          (3 言語)
  github.com/opaopa6969/tramli-appspec  (Java)
  github.com/opaopa6969/tramli-sdd      (Java, 未実装)
  github.com/opaopa6969/tramli-reactive (存在しない)
```

ユーザーから見ると **家族という集合が見えない**。4 つのリポジトリがバラバラにあるだけ。マニフェストは集合を宣言するが、コード実体は離散している。

**👤 今泉**: えっと、**「集合が見える」ってどういう状態**ですか？ 具体的に。

**🎨 深澤**: 例えば `github.com/opaopa6969/tramli-family` という親があって、
```
tramli-family/
  manifest.md
  decisions/
  sessions/
  core/           ← tramli 本体
  appspec/
  reactive/
  sdd-archive/
  tools/
    ci-check-manifest-dd-sync.sh
```
みたいな形なら、訪れた瞬間に「家族とはこれ」と掴める。手に取れる。

**🔬 ハレル教授**: 利点は理論的にも明確です。monorepo にすると:

```
1. 依存ツリー制約 (DD-044 Rule 1) の CI チェックが同一 repo で完結
2. マニフェスト ⇄ DD 同期 (Rule 2) のスクリプトが単一 CI に置ける
3. cross-member 変更 (tramli API 変更が appspec に波及) が atomic commit
4. tramli-sdd 吸収作業 (TramliStyleFlowValidator 書き換え) が 1 PR で済む
5. family-wide verification の実装場所が自然に決まる
```

**🧮 ミルナー**: 再現性の観点から言えば、**monorepo は hermetic build の前提** です。Nix flake や Bazel workspace は repo を単位とする設計。**家族全体の再現性を CI で保証するなら monorepo が圧倒的に楽**。

**☕ ヤン**: *紅茶を飲み干す。* 欠点は:

```
1. tramli (3 言語) + appspec (Java) の build system 混在
2. 各メンバーの独立リリースが複雑化
3. 外部ユーザーが tramli 単体で使いたいとき、余計なものがついてくる
4. 移行コスト (既存 repo 4 つの統合)
5. monorepo ツール (Nx / Bazel / Turborepo) の学習コスト
```

**👤 今泉**: えっと、**ちょうど 3 軸のトレードオフと同じ構造** ですね。
- monorepo の利点 = 検証可能性 + 再現性 (repo 次元)
- monorepo の欠点 = 独立性 + 単純性 の喪失

**🔬 ハレル教授**: 今泉君の整理が正しい。**monorepo は 3 軸の repo 次元への射影**です。repo を統合することは、家族の検証可能性・再現性を強化する代わりに、個別メンバーの独立性を犠牲にする。

**🎨 深澤**: ここで **階段状の移行** を提案します。
```
Step 1 (低コスト): umbrella repo を新設し、マニフェスト / DDs / sessions / CI だけ置く
                   各個別 repo は submodule 参照（または単に URL リンク）
Step 2 (中コスト): tramli-reactive を新規に umbrella 内で作る（既存 repo 統合は不要）
Step 3 (高コスト): 既存 repo を統合（必要があれば）
```

**いきなり全統合しない**。段階的に家族の実体を umbrella に集積していく。

**☕ ヤン**: *頷く。* *これは受け入れられる。* Step 1 はほぼ 0 コスト。Step 2 は新規なので移行なし。Step 3 は必要性が出てからで良い。

**🧮 ミルナー**: 理論的にも健全です。**段階的 bootstrapping** は多くの successful system が採用した戦略です。

→ アイデア: **umbrella repo (`tramli-family`) を新設** — マニフェスト / DDs / sessions / CI / 依存図だけ。個別 repo は当面維持
→ アイデア: **tramli-reactive は umbrella 内で新規作成** — 既存 repo の統合コストなしに家族の実体を増やせる
→ アイデア: **段階的統合を許容するが強制しない** — Step 3 (既存 repo 統合) は必要性が発生してから判断
→ Gap 発見: **umbrella 新設時の既存 DGE sessions / DDs の移行先が未決** — 現在 tramli repo 内 `dge/` にある。umbrella に移すか、tramli 内に残すか設計が必要

## Scene 5: 統合診断 — 家族の最終形

**先輩（ナレーション）**: 5 人が統合所見を述べる。

---

**🔬 ハレル教授**: 統合診断です。

```
理想形の再定義:
  ❌ 家族が「あらゆる部分」を書く全能フレームワーク
  ✅ 家族が主要領域を覆い、未覆領域は外部委譲として明示。盲点ゼロ。
  ✅ 検証可能 + 宣言的 は家族が担う。再現性は repo 次元（monorepo / Nix）に委譲。

移行戦略:
  Step 1: umbrella repo (tramli-family) 新設、文書と CI を集約
  Step 2: tramli-reactive は umbrella 内で新規
  Step 3: 既存 repo 統合は必要性に応じて判断
```

**🧮 ミルナー**: 理論的に健全。先行研究の教訓（MDA 挫折 = 3 軸同時追求）を回避し、関心事の分離（3 軸 = 3 レイヤー）を適用している。

**🎨 深澤**: *静かに言う。* 美しい形です。家族の集合が見える形になり、盲点ゼロを目指せる。**ユーザーから見て「手に取れる家族」** になる。

**☕ ヤン**: *紅茶を飲み干す。* ヤン的に削ると、**重要なアクションは 2 つ** です。
```
1. umbrella repo の新設 (Step 1)
2. マニフェスト本文執筆時に「家族のカバレッジ表」を盛り込む
```
Step 2/3 やカバレッジ強度の詳細は、この 2 つが済んでから判断すればよい。

**👤 今泉**: えっと、まとめると:
- 最終形 = 「検証可能 + 宣言的」を家族全体でカバー、盲点ゼロ、再現性は外部委譲
- monorepo は umbrella repo として Step 1 から始める（段階的）
- マニフェスト本文に「家族のカバレッジ表」を追加する

で正しいですか？

**☕ ヤン**: *頷く。* 正しい。**設計は決まった**。

→ アイデア: **最終形を「盲点ゼロの家族 + 3 軸の関心事分離」と定義** — マニフェスト冒頭の期待値調整に反映
→ アイデア: **Step 1 (umbrella repo 新設) を次の具体アクションとする**
→ アイデア: **umbrella repo 名は `tramli-family`** — シンプルで目的明示

## 決定事項（この session の合意）

- **最終形の再定義**: 家族が「あらゆる部分」を書くのではなく、**盲点ゼロ + 3 軸の関心事分離**
  - 検証可能 + 宣言的 = 家族が担う
  - 再現性 = repo 次元（monorepo / Nix）に委譲
- **monorepo は採用する**が、段階的に:
  - **Step 1**: umbrella repo `tramli-family` 新設 — 文書 / CI / 依存図のみ
