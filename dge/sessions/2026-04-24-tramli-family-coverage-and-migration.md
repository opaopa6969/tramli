## Scene 1: G11 カバレッジ表の粒度 — 「領域をどう分類するか」

**先輩（ナレーション）**: 最も具体的な Gap から。「主要領域」を分類する粒度を決めなければ、カバレッジ表が作れない。

---

**🎩 千石**: 粒度の選択肢を列挙します。

```
粒度 A (粗): 3 階層
  ドメイン層 / インフラ層 / UI 層

粒度 B (中): 7 階層（前セッションで提案）
  UI / 通信 / 永続化 / 認可 / 配置 / 監視 / 配線

粒度 C (細): 15+ 階層
  UI レンダリング / UI 状態管理 / フォーム / 検証 / 認証 / 認可 /
  セッション / メッセージング / RPC / ストレージ / マイグレーション /
  設定 / 配置 / 監視 / ロギング / トレース / ...
```

**🎨 深澤**: *ゆっくり。* 粒度 C は細かすぎて **手に取れない**。粒度 A は粗すぎて **実際の判断に使えない**。粒度 B が中間ですが、これでも 7 列の表は視覚的に重い。

**☕ ヤン**: *紅茶を置く。* マニフェスト 2 ページ以内 (DD-044) に収めるなら、**表は 5-6 行が限界** です。粒度 B でも多い。削るとしたら:

```
統合案 (粒度 B-2):
  ドメイン・ワークフロー (tramli/appspec が主戦場)
  通信・セッション     (tramli-reactive 想定領域)
  データ永続化         (外部委譲が多い領域)
  UI・入力            (外部委譲だが一部 appspec の Projection で対応)
  運用・配置          (完全に外部委譲)
```

5 行なら 2 ページに収まります。

**👤 今泉**: えっと、**「粒度 B-2」って 5 行ですが、読者によって「5 行すぎる」「足りない」が分かれませんか？** 別の読者には 10 行欲しい、という状況にどう対応します？

**🔬 ハレル教授**: 今泉君の指摘に対する理論的回答: **階層化する**。

```
階層構造の表:
  Level 1 (マニフェスト): 粒度 B-2 (5 行)
  Level 2 (付録または別文書): 粒度 B (7 行)
  Level 3 (個別メンバーの README): 粒度 C (必要に応じて)
```

読者は自分の関心の深さに応じて潜ります。マニフェスト本体は 5 行でシンプルに保ち、深掘りは委譲。

**🎩 千石**: 賛成します。**マニフェストは Level 1 のみ**。Level 2/3 へのリンクで拡張性を確保します。

→ アイデア: **マニフェスト本体のカバレッジ表は粒度 B-2（5 行）** — ドメイン / 通信 / 永続化 / UI / 運用
→ アイデア: **階層化戦略**: マニフェスト = Level 1、付録 = Level 2、個別 README = Level 3
→ Gap 発見の解消: **G11 粒度の問題は「階層化」で解消**

## Scene 2: G11 カバレッジ表の更新メカニズム — 「誰がいつ更新するか」

**先輩（ナレーション）**: 粒度が決まったので、次はライフサイクル。表は時間と共に古くなる。

---

**🎩 千石**: 表の更新には **3 つのトリガー** があります。

```
T1: 新しい家族メンバーが追加された
    → 列が1つ増える

T2: 既存メンバーの担当範囲が変わった
    → セルが更新される

T3: 外部委譲先（代表例）のツールが変わった
    → セルの委譲先記述が更新される
```

T1/T2 は DD を経るべき変更です（DD 採番時に同時にカバレッジ表を更新）。T3 は頻度が高いので、DD なしでの軽量更新を許容すべき。

**☕ ヤン**: T3 の軽量更新、具体的には？

**🎩 千石**: マニフェストの脚注レベルで更新する。例: 「UI 委譲先: React / Swing（代表例、2026-04-24 時点）」。日付を付記することで、古くなっても混乱しない。

**🎨 深澤**: *頷く。* 日付付記は美しい解です。読者は「ああ、これは 1 年前の状態ね」と自己判断できる。文書が古くなっても **壊れない** 。

**👤 今泉**: えっと、**T1 (新メンバー追加) の判断基準**って何ですか？ 誰が「新しい家族メンバーだ」と決めるんですか？

**🔬 ハレル教授**: 家族マニフェストに **「新メンバー追加の基準」** を付録として持つべきです。DD-044 で言及した 3 精度ルールを満たすこと + α として:

```
新メンバー追加の基準 (提案):
  1. 独立パッケージとして build / test できる
  2. 依存ツリーがルート (tramli) を向いている (DAG)
  3. 既存メンバーの責務と重複しない (3 章のマトリクスで重複セルが出ない)
  4. 宣言的 DSL または型付き API を提供する (家族の美学を守る)
  5. 提案 DGE session で承認される
```

**☕ ヤン**: *微笑む。* 5 番目が効きます。**DGE プロセスを家族の「門番」にする**。

→ アイデア: **カバレッジ表更新は 3 トリガー分類** — T1/T2 は DD を経る、T3 は軽量更新 + 日付付記
→ アイデア: **新メンバー追加基準をマニフェスト付録に明記** — 5 項目、うち最後は「DGE session で承認」
→ アイデア: **委譲先ツールには日付を付記** — `React (代表例、2026-04-24 時点)` という形式で「古くなっても壊れない」ドキュメント設計

## Scene 3: G10 — 3 軸独立性をマニフェストにどう書くか

**先輩（ナレーション）**: 抽象度が高い Gap。深澤さんが読者体験の観点から切り込む。

---

**🎨 深澤**: 「3 軸は独立」という事実を書くより、**「家族が担うのは 2 軸、残り 1 軸は repo 構造に委譲」** という **実装的な事実**を書く方が読者に伝わります。

```
悪い例 (抽象的):
  「検証可能・再現性・宣言的の 3 軸は独立次元であり、
    全てを同一レイヤーで満たすことは理論的に不可能です」
  → 読者: 何を言ってるか分からない

良い例 (具体的):
  「tramli 家族が担うのは『検証可能』と『宣言的』の 2 軸です。
    『再現性』は monorepo 構造（Nix / Bazel 等）に委譲します。
    3 軸を同じレイヤーで満たそうとした過去の試み（MDA/UML 等）は
    挫折しており、家族はこれを回避します」
```

**🔬 ハレル教授**: 賛成です。抽象的主張を避け、**「誰が何を担うか」** に落とす。MDA/UML の参照は学術的にも正確です。

**🎩 千石**: この表現なら **マニフェスト 1 章（これは何か）の 1 段落** として入ります。わざわざ独立した章を立てる必要がない。冒頭で短く触れて、詳細は非ゴール章に回す。

**☕ ヤン**: *そう。* ヤン的には、**抽象的な 3 軸理論はマニフェストに書かない**。書くとマニフェストが肥大化する。書くのは「家族が担う 2 軸」と「repo に委譲する 1 軸」という **事実だけ**。

**👤 今泉**: えっと、じゃあ **MDA/UML の挫折についてはどこに書く** んですか？ 非ゴール章でしたよね？

**🎩 千石**: 非ゴール章の冒頭 1 行で触れます:

```
## 5. 非ゴール
本ファミリーは 3 軸（検証可能・再現性・宣言的）を同じレイヤーで同時に
満たすことを目指しません（MDA/UML の挫折の教訓）。

### 5.1 ファミリー全体の非ゴール
  ...
```

1 行で参照するだけ。詳細の歴史分析はしない。マニフェストの役割外。

→ アイデア: **マニフェスト 1 章で「家族が担う 2 軸、委譲する 1 軸」を具体的に書く** — 抽象的な独立性理論は書かない
→ アイデア: **非ゴール章の冒頭 1 行で MDA/UML 挫折を参照** — 詳細分析はしない、事実のみ
→ Gap 発見の解消: **G10 は「抽象でなく具体」で書くことで解消**

## Scene 4: G12 — 既存 DGE 文書の移行先

**先輩（ナレーション）**: 最後は実務的な判断。umbrella repo 新設時、既存の `dge/sessions/` `dge/decisions/` をどうするか。

---

**☕ ヤン**: 選択肢を並べます。

```
M1: 全部 tramli repo に残す
    → 家族の文書が片メンバーの repo にある違和感

M2: 全部 umbrella repo に移す
    → 既存リンクが全部壊れる。セッション内のリンク修正コスト大

M3: 分離する (tramli 固有 vs 家族全体)
    → 判断コストあり。どちらに属するか議論

M4: 双方に参照を置く
    → 文書の重複リスク
```

**🎩 千石**: M2 の「リンクが壊れる」を詳しく:

```
現在:
  dge/sessions/2026-04-24-tramli-dataflow-mode.md
    → ../decisions/DD-042-tramli-exemplar-completion.md

もし umbrella に移すと:
  tramli-family/sessions/2026-04-24-tramli-dataflow-mode.md
    → ../decisions/DD-042-tramli-exemplar-completion.md  (相対パス維持)

相対パスなら壊れない。ただし sessions 間の相互参照は全てチェック必要。
```

**🎨 深澤**: *考え込む。* ユーザー体験の観点から見ると、**今の DGE 文書は「家族全体の設計史」**です。tramli 本体の機能文書ではない。umbrella に移す方が **意味的に正しい** 。

**👤 今泉**: えっと、**過去の session を読む人は誰ですか？** tramli の実装者？ コントリビューター？ それとも「家族とは何か」を知りたい人？

**🔬 ハレル教授**: 今泉君の問いは核心的です。読者を仮定すると:

```
R_past: tramli 本体の歴史的議論を見たい (実装者)
  → tramli repo に残すべき

R_family: 家族全体の設計議論を見たい
  → umbrella に移すべき

R_mixed: 両方を跨ぐ DGE もある (DD-042/043 など)
```

現実には **混合**です。だからどちらか一方は不正確。

**☕ ヤン**: *紅茶を飲む。* **時期で分ける**のはどうです？

```
M5 (時系列分割):
  2026-04-24 以前: tramli repo に残す (tramli 単体の歴史)
  2026-04-24 以降: umbrella repo (家族全体の議論)

  理由: 本 session から家族モデルが正式採用されたため、
       2026-04-24 を境界とするのは意味論的にも自然
```

**🎩 千石**: それは美しい解決です。**過去を動かさない** = 既存リンクが壊れない。**未来を新居に置く** = 意味論的に正しい。

**👤 今泉**: えっと、**2026-04-24 当日の session は境界ですが、どっちに行きますか？**

**🎩 千石**: 2026-04-24 の本日の 6 session は **家族モデル議論そのもの**。umbrella に移す方が正しい。

**🎨 深澤**: *頷く。* 2026-04-24 は umbrella repo の **創設文書** になります。誕生日。美しい。

**☕ ヤン**: *微笑む。* この案だと **移行コストは本日分だけ** です。過去はそのまま。実装コスト極小。

→ アイデア: **時系列分割戦略（M5）** — 2026-04-24 以前は tramli repo に残し、以降は umbrella
→ アイデア: **本日 2026-04-24 の 6 session は umbrella repo の創設文書** として移行
→ アイデア: **tramli repo の `dge/` に「家族全体の議論は umbrella repo へ」という案内を残す** — 読者が迷子にならないように
→ Gap 発見の解消: **G12 は時系列分割で移行コスト最小化しつつ意味論を保つ**
