## Scene 1: G9 の本質を詰める — 「何が双方向で、何がツリーを破るか」

**先輩（ナレーション）**: 最も重い Gap から。ツリー制約を宣言した直後に双方向接続が出た矛盾。千石がまず「双方向」の実体を定義する。

---

**🎩 千石**: G9 を厳密に分解します。「双方向」と言ったとき、2 つの意味があります。

```
A. 型依存の双方向
   tramli-appspec が tramli-reactive の型を使い、
   tramli-reactive も tramli-appspec の型を使う
   → パッケージ間循環依存。コンパイル不可能。

B. データフローの双方向
   tramli-appspec が定義した TaskSpec の外部 I/O を
   tramli-reactive の Channel に bind する
   → 実行時にメッセージが双方向に流れる。
     ただし型依存は片方向でありうる。
```

前回 session で私が書いた `ReactiveChannelBinding: TaskSpec の外部 I/O を Channel に bind` は **B（データフロー双方向）** であって、**A（型依存）ではない**。

**☕ ヤン**: *紅茶を置く。* つまり**矛盾ではない**ということですね？ 型依存がツリー（appspec → reactive の一方向）なら、実行時のメッセージが双方向に流れても、パッケージ構造はツリーを保てる。

**🔬 ハレル教授**: *頷く。* これは圏論でも普通の話です。圏の射は一方向ですが、**対象間のデータは射に沿って往復できます**。プログラミング言語で言えば、関数の引数/返り値は型的には一方向依存ですが、実行時には双方向にデータが動きます。

**👤 今泉**: えっと、素人質問ですけど、**「ツリー制約」って何を禁じているんですか？** 型依存？パッケージ依存？データフロー？マニフェストに書くときに混乱しそうです。

**🎩 千石**: 今泉さんが鋭い。**ツリー制約の対象を明示する** べきです。

```
ツリー制約（明示化）:
  対象: パッケージ import 依存（Maven / npm / Cargo の依存グラフ）
  禁止: 循環依存
  許可: 実行時メッセージの双方向、callback、event publish-subscribe
```

**🎨 深澤**: *静かに言う。* デザインの観点から見ると、**「ツリー」という単語は双方向性と両立しにくい印象を与える**。マニフェストを読んだ人は「データも片方向しか流れないのか」と誤読します。より正確には **「依存ツリー」** と呼ぶべきです。

→ Gap 発見の解消: **G9 は設計矛盾ではなく用語の曖昧さが原因**だった。「ツリー制約」の対象を「パッケージ import 依存」に限定すれば、データフロー双方向とは両立する。

→ アイデア: **マニフェストでは「依存ツリー制約」と明示的に書く** — 対象は import 依存グラフのみ。データ流 / event / callback は対象外と明記。

→ アイデア: **依存ツリー構造の Mermaid 図と、データフロー双方向の Mermaid 図を 2 枚に分ける** — 同じ図に混ぜると読者が混乱する。依存図は常に DAG、データ流図は自由。

## Scene 2: G8 — マニフェストと DD の同期メカニズム

**先輩（ナレーション）**: ドキュメントの整合性問題。ハレル教授が最適解を提案する。

---

**🔬 ハレル教授**: マニフェストと DD の関係を整理します。DD は **時間の連続** で、マニフェストは **現時点のスナップショット**。常に同期させるのは理論的には重い。

```
DD:         DD-001 → DD-002 → ... → DD-043 → DD-044 → ...
マニフェスト: (DD-043 時点のスナップショット)
```

同期メカニズムの選択肢:

```
S1. マニフェスト本文に DD 番号を埋め込む
    例: 「tramli は exemplar として完結させる [DD-042]」
    → 利点: どの DD が根拠か一目瞭然
    → 欠点: DD が supersede されるとリンクが古くなる

S2. マニフェストの末尾に「根拠 DD 一覧」を置く
    例: 「本マニフェストの根拠: DD-042, DD-043」
    → 利点: 集約されていて整合確認が容易
    → 欠点: 本文とのリンクが弱い

S3. マニフェストを Generated Document にする
    → DD から自動生成
    → 利点: 常に最新
    → 欠点: 人間が書く余地が消える、微妙なニュアンスが失われる
```

**☕ ヤン**: *紅茶を飲む。* S3 は過剰です。マニフェストは人間が読む文書なので、自動生成すると質が死ぬ。**S1 + S2 の併用**でいいのでは。

**🎩 千石**: 同意します。S1 で本文中の主張に DD を付し、S2 で末尾に一覧を置く。**ただし DD 側にも「このマニフェストから参照されている」と逆リンクを貼る**必要があります。

**👤 今泉**: えっと、**逆リンクを手で管理すると絶対ズレますよね？** DD を書くときに「マニフェストに追加」を忘れる。**自動化できる部分と手動の部分を分けて**書くのが良いのでは。

**🎨 深澤**: *そう思う。* 僕も同意です。ドキュメントの整合性は「気合で維持する」では必ず崩れる。**どこかで自動チェックが要ります**。

**🔬 ハレル教授**: 実用的な妥協案:

```
自動化: CI で「マニフェストに現れる DD 番号」と「DD 側の逆リンク」が
        一致するかをチェックする簡易スクリプト

手動:   マニフェスト本文の記述
        DD の Rationale
```

**☕ ヤン**: *これで十分です。* 軽量 CI チェックなら導入コスト低い。**マニフェスト側に DD 番号を書くのは必須、逆リンクは CI で警告**。これで崩壊しません。

→ アイデア: **マニフェストに DD 番号埋め込み（S1） + 末尾根拠一覧（S2）** を両方採用
→ アイデア: **マニフェスト ⇄ DD の逆リンク整合を CI で自動チェック** — 簡易 node/bash スクリプトで「マニフェスト内の DD-NNN 番号」と「DD 側の参照される側の一覧」を突合し、不一致を警告
→ Gap 発見の解消: **G8 は CI 軽量チェック + 明示ルール**で実運用可能

## Scene 3: G7 — 「全部書くためではない」期待値調整

**先輩（ナレーション）**: 読者の期待値を誤らせないための書き方。深澤さんが UX 視点から切り込む。

---

**🎨 深澤**: 期待値調整は **冒頭の 3 行** で決まります。読者が最初の段落を読んだ時点で「これは何ではないか」を理解させる。

```
悪い例（肥大化を招く）:
  「tramli ファミリーは、アプリケーションを宣言的に定義する
    統合フレームワークです」
  → 読者: 「全部できるんだ」と誤読

良い例（期待値調整済み）:
  「tramli ファミリーは、アプリケーションの**一部分を構造的に検証できる**
    独立した宣言系の集合です。UI レンダリング、通信プロトコル、
    インフラ配置などは別ツールに委譲します」
  → 読者: 「検証できる部分に限定される」と正しく理解
```

**👤 今泉**: えっと、**「別ツールに委譲」と書いたら、「じゃあどのツール？」と聞かれませんか？** 親切に答えると肥大化しますし、答えないと不親切。

**🎩 千石**: 今泉さんの懸念は正しい。**「代表例」を 1 つだけ挙げる** のが良い妥協点です。

```
UI レンダリング: 各言語の標準 UI フレームワーク（React, Swing 等）に委譲
通信プロトコル:  将来の tramli-reactive（未実装）または既存の Reactive ライブラリ
インフラ配置:    Terraform / Pulumi / Kubernetes manifest 等に委譲
```

代表例だけ示し、選定はユーザーに委ねる。**マニフェストが選ぶべきではない**（ユーザーの技術選定を縛らない）。

**🔬 ハレル教授**: 理論的に言えば、これは **「定義の負論理 (negative definition)」** です。「何であるか」より「何でないか」で定義する方が、対象を正確に特定できる場合があります。数学でも集合の差集合でよく使われます。

**☕ ヤン**: *紅茶を置く。* 深澤さんの「冒頭 3 行で決める」が強いです。**冒頭 3 行 + 非ゴール章（5 章）の両方で期待値を調整**する。冒頭で「これではない」を示し、非ゴール章で詳細を列挙。読者は 2 度聞かされる。

→ アイデア: **マニフェスト冒頭 3 行で「ファミリーは〜ではない」を明示** — negative definition。読者が最初の段落で期待値を正しく設定できる。
→ アイデア: **「別ツールに委譲する領域」を代表例 1 つだけ挙げる** — ユーザーの技術選定を縛らず、しかし「どこに行けばいいか」を示す。
→ Gap 発見の解消: **G7 は冒頭 3 行 + 非ゴール章の二重強調**で解消

## Scene 4: 統合診断 — 3 Gap 潰し込み

**先輩（ナレーション）**: 5 人が統合所見を述べる。

---

**🔬 ハレル教授**: 3 Gap の性質を整理します。

```
G7: UX/表現の問題       → 書き方の工夫で解決

G9: 用語の曖昧さの問題    → 「依存ツリー」への精密化で解消
```

**3 つとも根本的な設計変更を要さない**。マニフェスト骨子自体は強固です。

**🎩 千石**: 同意します。3 Gap は「マニフェストの書き方の精度」の問題であって、構造の問題ではない。骨子を変える必要はない。

**🎨 深澤**: 僕も同意です。**骨子は既に手に取れる形**になっています。これ以上抽象を積むのは蛇足。

**👤 今泉**: えっと、じゃあ **次は実際にマニフェストを書き始められる** ということですか？ 設計フェーズは終わりに近づいている。

**☕ ヤン**: *紅茶を飲み干す。* *微笑む。* そうですね。**設計を詰めるフェーズはほぼ完了**です。あとは本文を書くだけ。ただしユーザーは「実装には入らない」と明言したので、マニフェスト本文執筆も **次のセッション以降** に回す判断もあり得ます。

**🔬 ハレル教授**: 理論家として一言。**設計は完璧ではなく「十分に良い」で止める**のが正しい。完璧を追うと無限に議論が続きます。G7/G8/G9 は「十分に良い」に達しています。

→ アイデア: **マニフェスト本文執筆は次セッションから** — 本 session で設計フェーズは完了とする
→ アイデア: **本 session の合意を DD-044 として記録** — 「ツリー制約は依存 import のみ」「マニフェスト ⇄ DD は S1+S2+CI」「期待値は冒頭 3 行 + 非ゴール」の 3 点

- **「ツリー制約」は「依存ツリー制約」と精密化**。対象は import 依存グラフのみ。データフロー / event / callback は対象外

- **期待値調整は冒頭 3 行の negative definition + 非ゴール章** の二重強調
- **委譲先ツールは代表例 1 つのみ**（ユーザーの選定を縛らない）
- **設計フェーズは本 session で完了**。マニフェスト本文執筆は次セッション以降
- **本 session 合意を DD-044 として記録**（次の選択肢）
