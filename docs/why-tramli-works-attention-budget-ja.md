# なぜ tramli は効くのか —— アテンション・バジェット

あなたにはバジェット（予算）がある。お金ではない——**注意力**の予算だ。

コードを読むたびに、脳はこの予算を消費する。変数名を読む：少し減る。その変数がどこから来たのか、3ファイル・400行をまたいで追跡する：大きく減る。予算が尽きると、見落としが始まる。バグが忍び込む。

これは比喩ではない。脳が実際にそう動いている。そして驚くべきことに、AI 言語モデルもまったく同じ仕組みで動いている。tramli はこのたった一つの洞察を中心に設計されている。

---

## 脳には RAM の上限がある

やってみよう。この数字を覚えてほしい。

> 7, 2, 8, 4, 1, 9, 3

目を閉じて、繰り返せるだろうか。

ほとんどの人がワーキングメモリに同時に保持できるのは約7項目。1950年代から研究されていて、心理学者ジョージ・ミラーはこれを「マジカルナンバー7±2」と呼んだ。

500行の関数を読んでいるとき、あなたは単にコードを読んでいるのではない——**頭の中で状態を保持している。** 「えーと、`returnTo` は38行目でセットされて…167行目でチェックされるけど `fwdProto` が null じゃないときだけで…`fwdProto` は HTTP ヘッダーから来るんだけどそれは Traefik がセットするもので、でもミドルウェアが動いたときだけで…」

7項目どころの話じゃない。予算は吹き飛んでいる。何かを見落とす。

---

## LLM にもまったく同じ問題がある（文字通り）

Claude や GPT のような AI 言語モデルは **アテンション** という仕組みを使う。モデルがコードを読むとき、すべてのトークン（おおまかに言えば、すべての単語や記号）が文脈を理解するために他のすべてのトークンを「見る」。これが Transformer アーキテクチャの中核だ。

ここがポイント：アテンションには**レイヤーごとの固定予算**がある。モデルはコンテキストウィンドウ内のすべてのトークンに注意力を分配する。読むコードが増えれば増えるほど、注意力は薄く広がる。

```
  50行の FlowDefinition → アテンションが集中    → モデルは深く理解する
1800行のハンドラ        → アテンションが希薄化   → モデルは繋がりを見失う
```

これは設計上の欠陥ではない——数学的な性質だ。アテンションの重みはすべてのポジションにわたって合計1.0にならなければならない。ポジションが増える＝1ポジションあたりの重みが減る＝個々の接続のシグナルが弱くなる。

だから LLM が1800行のハンドラを読んで、384行目でセットされた `return_to` が1204行目で消費されていることを見落としたとき、それは「ハルシネーション」ではない。人間のプログラマーがそうなるのと同じように、アテンション・バジェットを使い果たしているのだ。

## 手続き型コードがバジェットに何をするか

ちょうど起きた実際のシナリオがある（名前は簡略化）：

```
384行目:   returnTo = (fwdProto != null ? fwdProto : "http") + "://" + fwdHost + fwdUri
404行目:   proto = fwdProto != null ? fwdProto : "http"
650行目:   ctx.cookie("__volta_session", sessionId, ...)  // Secure フラグは？
890行目:   ctx.redirect(loginUrl + "?return_to=" + returnTo)  // まだ使える？
1204行目:  window.location.href = result.redirect_to || '/console/'  // return_to はどこへ？
```

バグを見つけるには、これらを**すべて同時に**頭に入れておく必要がある：

1. `returnTo` がどこで作られたか（384行目）
2. `fwdProto` が null のときのデフォルト値（384行目 —— "http"）
3. `fwdProto` がどこから来るか（Traefik ForwardAuth ヘッダー）
4. Traefik が `X-Forwarded-Proto` を渡すかどうか（ミドルウェア設定次第）
5. ログインページが POST を通じて `returnTo` を保持するかどうか（してない）
6. Cookie に `Secure` フラグがあるかどうか（`isSecure()` 次第、それは Traefik の接続方式次第）

800行以上、3つの異なるシステムにまたがる6つの事実。どんな人間にも、どんな LLM にも、これを確実に保持することはできない。

**結果：** AI エージェントが14分間ぶっ続けでデバッグした——サーバーに SSH して、設定を読み、コードを修正し、巻き戻し、また修正し——バグを一つずつ追いかけ、修正するたびに次の問題が露呈した。人間でも同じことをするだろう。ただもっと遅いだけだ。

## tramli がバジェットに何をするか

tramli は同じロジックを再構成し、各部品が**何を必要とし、何を提供するか宣言させる：**

```java
// LoginRedirectInit
requires: { RequestOrigin, AuthConfig }
produces: { LoginRedirect }

// SessionCreator
requires: { ResolvedUser, RequestOrigin, AuthConfig }
produces: { SessionCookie, FinalRedirect }
```

さっきの6つの質問が、すべてローカルな答えを持つようになる：

| 質問 | 手続き型（1800行を読む） | tramli（1つの Processor を読む） |
|------|--------------------------|----------------------------------|
| `returnTo` はどこから来る？ | 384行目から追跡 | `LoginRedirectInit.produces(LoginRedirect)` |
| セッション作成時に `returnTo` は使える？ | 384〜1204行目を読む | `build()` が検証済み —— 使える |
| URL スキームは？ | 3つのシステムをまたいで `fwdProto` を追跡 | `RequestOrigin.scheme` —— フィールド1つ |
| Cookie に `Secure` はある？ | 650行目を探して `isSecure()` を追跡 | `SessionCookie.create(origin)` —— メソッド1つ |

**各質問のコストが、ワーキングメモリ6項目ではなく1項目になる。** 予算内に収まる。バグは起きない。

## 「読まなくてよかったもの」の原則

ここが最も直感に反する洞察だ：**tramli の価値は、読ませるものにあるのではない——読み飛ばせるものにある。**

1800行のハンドラでは、すべての行が暗黙のコンテキストだ。400行目を変えると1200行目が壊れるかもしれない。全部読まないとわからない。だから全部読むしかない。

tramli では、`StateProcessor` は**閉じた単位**だ。入力は宣言されている（`requires`）。出力も宣言されている（`produces`）。`SessionCreator` を修正するとき、`TokenExchange` も `LoginRedirectInit` も読む必要がない。それらが互いに影響することは不可能——コンパイラと `build()` が保証している。

```
手続き型:  1800行 × 「関係あるかも」 = 1800行を読む
tramli:    50行の FlowDefinition + 必要な1つの Processor 30行 = 80行を読む
```

アテンションコストの95%削減。情報を圧縮したのではない——**コードの95%が自分のタスクに無関係であることを証明した**のだ。

## なぜ人間にも AI にも効くのか

この並行性は偶然ではない。構造的なものだ：

| | 人間の脳 | LLM のアテンション |
|---|---|---|
| 容量 | ワーキングメモリに約7項目 | レイヤーごとの固定アテンション予算 |
| 失敗パターン | 「800行前に `returnTo` がセットされたの忘れてた」 | 384行目へのアテンション重みが低すぎて1204行目と繋がらない |
| 効くもの | 局所性——関連するものが近くにある | 局所性——関連するトークンが近くにある |
| 害になるもの | 1000行以上にわたるグローバル依存 | 長距離の依存がアテンションを希薄化 |

tramli は**グローバル依存**（384行目が1204行目に影響する）を**ローカルな契約**（隣接する Processor の `requires`/`produces`）に変換する。人間には効く、なぜならワーキングメモリに収まるから。LLM にも効く、なぜなら関連するトークンがコンテキストウィンドウ内で近くにあるから。

だからこそ LLM が tramli コードを安全に**生成**できる：仮に間違った遷移をハルシネートしても、`build()` が即座にリジェクトする。フィードバックループは：生成 → コンパイル → `build()` → 修正。14分のデバッグセッションは要らない。「サーバーに SSH して tail -f」も要らない。

## 3つの保証

tramli は3つのことを**構造的に不可能**にする——「起きにくい」でも「テストで捕まる」でもなく、不可能に：

**1. データの欠落**
```java
// SessionCreator が LoginRedirect を必要としているのに
// 何も produces していなければ、コードが走る前に build() が失敗する：
//
// "Processor 'SessionCreator' at SESSION_CREATED requires LoginRedirect
//  but it may not be available"
```

**2. 不正な遷移**
```java
// ステートは enum。"COMLETE" のようなタイポはコンパイルエラー。
// auto 遷移のサイクルは build() の DAG 検証で捕まる。
// terminal ステートからの遷移は build() で捕まる。
```

**3. 古くなった図**
```java
// 図はコードそのもの。エンジンが実行する FlowDefinition から生成される。
// 古くなりようがない。
String mermaid = MermaidGenerator.generate(authFlow);
```

これらは「ベストプラクティス」ではない。**コンパイラが強制する不変条件**だ。覚えておくためにアテンション予算を使う必要がない——ツールチェインが覚えてくれる。

## まとめ

| | tramli なし | tramli あり |
|---|---|---|
| フローを理解するアテンションコスト | 1800行を読む | 50行の FlowDefinition を読む |
| 1ステップを変えるアテンションコスト | 全部読む（何か壊すかも） | 1つの Processor を読む（他を壊せない） |
| 「ここで `returnTo` は使える？」 | 800行を追跡 | `build()` が検証済み |
| データ欠落バグのデバッグ | 14分の SSH + ログ読み | コンパイル時の `build()` エラー |
| 人間に効く？ | — | ✓（7±2 のワーキングメモリに収まる） |
| LLM に効く？ | — | ✓（アテンション・バジェットに収まる） |

**tramli はあなたを賢くしない。問題を小さくする——あなたが既に持っているアテンション・バジェットに収まるほど小さく。**

---

*tramli = tramline（路面電車の軌道）。コードはレールの上を走る。軌道が敷かれた場所にしか行けない——だからこそ迷わない。*
